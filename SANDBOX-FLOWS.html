LCM-Sandbox — End-to-End Flows

Three trigger paths (direct CLI / AIDevOps UI / AIDevOps programmatic), their cleanup flows, failure paths, and the sandbox state machine. Mermaid sequence diagrams + flowcharts. Companion to SANDBOX-ORCHESTRATION.md and SANDBOX-DETAILED-FLOW.md.

Path A — Direct CLI (developer in terminal)

Who initiates: A developer, at their own workstation, opening a terminal and typing lcm-sandbox create .... No web UI, no AIDevOps involvement, no MCP back-channel.

When to use: Local development of consumer projects, manual ad-hoc sandbox runs, debugging the tool itself, smoke-testing before integration.

Auth: Developer's local ANTHROPIC_API_KEY env var. Observability: stdout JSON + structured logs to stderr. Cleanup: manual lcm-sandbox cleanup --sandbox-id .... No mid-run channel.

Path B — AIDevOps UI (developer clicks "Run plan")

Who initiates: A developer, in their browser, on the AIDevOps web UI, after reviewing an approved plan, clicks a "Run plan in sandbox" button.

What's different from Path A: AIDevOps backend invokes the CLI as a subprocess, mints a per-run MCP token, exposes a live channel back to the workflow engine, and stores artifacts in the AIDevOps DB instead of just printing JSON.

Auth: Per-run Anthropic key minted by AIDevOps + per-run bearer for the MCP channel. Observability: live progress + approval pings via WebSocket. Cleanup: still requires either manual click or PR-merge webhook.

Path C — AIDevOps programmatic (no human button-click)

Who initiates: Nobody, in real time. A workflow trigger fires: a GitHub webhook on PR open, a cron, an upstream event, a scheduled audit, etc.

What's different from Path B: No frontend involvement initially. Human-in-the-loop only happens if the workflow needs an approval (gated by risk classification). Otherwise the entire run is autonomous.

    alt risk == low
        Plan->>BE: auto-run
    else risk in {medium, high, critical}
        Plan->>FE: emit approval-needed event
        FE-->>Dev: notification (browser, slack, email)
        Dev->>FE: click approve
        FE->>BE: forward decision
        BE->>Plan: approval granted
    end

    BE->>CLI: spawn(lcm-sandbox create ... --mcp-endpoint ... --mcp-token-file ...)
    Note over CLI,Cont: (Same Phases 0-4 as Path B)
    Cont->>MCP: tools/list (Bearer)
    MCP-->>Cont: tool surface

    Note over Cont,MCP: Autonomous execution — no human watching real-time
    Cont->>MCP: report_progress (heartbeat every N seconds)
    Cont->>MCP: submit_intermediate_artifact (logs, diffs)
    MCP->>DB: persist
    end

    alt agent hits a destructive op flagged in plan
        Cont->>MCP: request_human_approval
        MCP->>FE: push approval needed
        FE-->>Dev: page on-call
        Dev->>FE: approve / deny
        FE-->>MCP: decision
        MCP-->>Cont: response
    end

    Cont->>MCP: notify_completion
    CLI->>BE: final artifacts
    BE->>DB: agent_artifacts + audit_events

- open PR on origin
- run consumer tests
- request code review

Auth: Same as Path B. Observability: async — humans see progress later via dashboards, paged only on approval requests or failure. Cleanup: PR-merge webhook fires the cleanup flow.

What happens when a phase fails. Different paths have different escape routes.

Key invariant: any failure after Phase 4 (container launched) still runs Phase 6 (artifact capture) to preserve whatever the agent did before the failure. This makes post-mortem possible even on partial runs.

Sandbox lifecycle state machine

Every sandbox run progresses through these states. State changes are persisted to agent_artifacts.status in AIDevOps DB for paths B and C; printed to stdout for path A.

stateDiagram-v2
    [*] --> Requested
    Requested --> Preflight: validated by AIDevOps risk gate (B, C) / immediate (A)
    Preflight --> WorktreePrep: phases 0 ok
    Preflight --> Errored: preflight fail
    WorktreePrep --> Sync: worktree created/reset
    WorktreePrep --> Errored: worktree fail
    Sync --> DockerPrep: in_sync / behind→rebased
    Sync --> Errored: ahead / diverged / rebase conflict
    DockerPrep --> Ready: image present + verified
    DockerPrep --> Errored: image missing / broken
    Ready --> Running: docker run + entrypoint
    Ready --> Errored: container failed to start
    Running --> AwaitingApproval: agent calls request_human_approval (B, C)
    AwaitingApproval --> Running: approved
    AwaitingApproval --> Errored: denied or approval timeout
    Running --> Capturing: container exits (success or timeout)
    Capturing --> Captured: artifacts persisted
    Capturing --> Errored: capture failed
    Captured --> AwaitingMerge: PR opened
    AwaitingMerge --> CleanedUp: PR merged -> webhook (B, C) / manual cleanup (A)
    AwaitingMerge --> AwaitingMerge: PR pending
    CleanedUp --> [*]
    Errored --> [*]: triage

What this doc does NOT cover (yet)

  • Multi-step plans — where one run feeds the next via worktree reuse. Implementation supports it (--existing-worktree flag); flow diagram TBD.
  • Concurrent runs against the same repo — multiple worktrees in .sandbox-worktrees/. Documented in architecture; flow diagram TBD.
  • Anthropic API failure handling — what happens if the model API is down mid-run. Out of scope for sandbox; agent layer concern.
  • The macos-app launcher — auto-launches the AIDevOps server. Not a user-facing entry point; documented in aidevops/macos-app/README.md.
