Agentic Sandbox Control Plane

How AIDevOps controls, observes, and reacts to work happening inside an agentic sandbox. Pairs with SANDBOX-ARCHITECTURE.md (what runs where) and SANDBOX-ORCHESTRATION.md (live transport).
Revision 2 (2026-06-19) — incorporates corrections from late-session design discussion: sparse checkout v1, no host-side POSIX enforcement, required_paths + read_exclusions, repo_kind, Hermes egress, topology-driven resolution, test mounts. Supersedes Revision 1 (2026-06-18).

Overview

The agentic sandbox accepts a plan from AIDevOps, executes it under bounded permissions, and reports status and events back continuously. AIDevOps can intervene (pause, cancel, answer the agent's questions); the sandbox can request (credentials, decisions, operator input). When the agent finishes, the sandbox emits a completion event with a manifest; AIDevOps then drives the next workflow step (testing, review, merge).

Key design choices:
  • AIDevOps never talks to the agent directly. It talks to Hermes in-container.
  • Two container mounts: a /workspace mount with sparse checkout for visibility plus ro/rw overlays for write scope; and a Hermes-only /control mount for plan-in and status-out.
  • Mount surface is the real restriction. Host-side POSIX permissions do not translate through Colima virtiofs to in-container enforcement and are not a defense layer.
  • Plan declares repo_kind (new vs existing) and required_paths (rw narrow) plus read_exclusions (broad ro by default for existing repos, with secrets carved out).
  • Hermes mediates all LLM egress and applies classification policy, which is what makes broad read scope safe in the existing-repo case.

Enforcement layers

Three real enforcement layers plus one audit layer. Each does something different; security relies on stacking them.

LayerWhat enforcesWhereBypass surface
1. Mount visibilitySparse checkout on host + selected bind mounts: paths not declared are physically absent from the containerHost worktree + Docker mount listContainer escape (very hard)
2. Docker bind mount modeLinux kernel VFS rejects writes on :ro mounts; allows on :rwLima VM kernelContainer escape
3. Hermes application-layer refusalHermes refuses agent tool calls outside declared paths and logs each refusalIn-container, mediates tool invocationsAgent invokes filesystem syscalls directly without going through Hermes
4. Container hardeningaiagent UID, --cap-drop=ALL, --no-new-privileges, --read-only root, noexec tmpfsVM kernel and DockerKernel CVE or container escape
Host-side POSIX is NOT a defense layer on this stack. Verified empirically on macOS + Colima 2026-06-19: chmod and chown applied on the host do not translate through virtiofs to in-container enforcement. The aiagent user reads and writes regardless of host-side mode bits. Any earlier reference to POSIX as defense-in-depth was incorrect and has been removed from this design.

Worktree lifecycle and mount selection

Creation

The worktree is created by lcm-sandbox Phase 1 (lcm_sandbox/core/worktree.py) after AIDevOps writes the plan to /control/plan/ and before docker launch. It is pinned to the source_revision the plan declares, so the agent works on a frozen snapshot regardless of what the developer has uncommitted in the parent repo.

Sparse checkout for visibility (v1)

After git worktree add, Phase 1 applies git sparse-checkout using the declared read scope minus read exclusions. Paths outside the resulting set are not physically present in the worktree directory. The agent cannot read them, even via prompt injection or tool misuse, because they do not exist in the filesystem the container sees. 

# inside the new worktree, before docker run
git sparse-checkout init --cone
git sparse-checkout set <declared read paths minus exclusions>

Mount overlay model

The whole sparse-checked-out worktree is bind-mounted at /workspace as :ro. Each rw subpath declared by the plan is then mounted on top as :rw: 

docker run \
  -v <worktree>:/workspace:ro \
  -v <worktree>/src/parser:/workspace/src/parser:rw \
  -v <worktree>/tests/parser:/workspace/tests/parser:rw \
  -v <control>:/control:rw \
  ...
Verify overlay behavior on Colima before relying on it. Docker bind mount overlapping is supposed to shadow per-subtree. The mount enforcement test plan below includes an explicit test (T1 and T2) for this on Colima. Fallback if overlaps misbehave: enumerate every directory and file individually as :ro or :rw.

Repo kind: new vs existing

The plan declares repo_kind. The two cases have fundamentally different threat models and different mount defaults.

Fieldrepo_kind: newrepo_kind: existing
Threat modelNothing to break yet; agent has latitudeFunctional code present; must not break what is outside declared scope
Default required_paths.rwProject root (or empty if not relevant)Narrow declared scope only, derived from plan intent
Implicit read scopeWhole worktree minus exclusionsWhole worktree minus exclusions (broad read for context)
Sparse checkout patternInclude everything except read_exclusionsInclude everything except read_exclusions
Hermes non-regression instructionInactiveActive. Agent is told do not break existing functionality outside declared scope; escalate via ask if scope insufficient.
Typical exit_criteriaTests pass, build passesTests pass, build passes, plus non-regression command verifies untouched code paths still behave
Failure mode if scope creepsLess criticalHermes refuses tool calls outside scope; agent must escalate

Why broad read is safe in the existing-repo case

Restricting reads would make the agent more likely to break things, not less, because the agent cannot understand context it cannot see (types, callers, sibling modules, conventions). Broad read is correct for the existing-repo case.

The exfiltration risk that broad read raises is mitigated by Hermes egress classification: every LLM call passes through Hermes, which applies the data classification policy (see aidevops/design/LLM-DATA-CLASSIFICATION-POLICY.md) on outbound payloads. Sensitive content (PII, secrets, credentials, tokens, customer data) is blocked, redacted, or requires operator approval before egress.

This is a layered defense:

  • Layer 1 — read_exclusions in the plan: the agent never sees secrets at all (they are not in the sparse-checked-out worktree).
  • Layer 2 — Hermes classification: anything that slipped in or appears intentionally in source code is gated at the egress point.

Topology-driven mount resolution

The plan does not have to enumerate every concrete path. Instead, it can reference logical scope (modules, include tests, include docs); the launcher reads the target repo's docs/REPO-TOPOLOGY.md and resolves logical scope to concrete required_paths deterministically. 

# plan.json fragment with logical scope
"scope": {
  "modules":       ["parser"],
  "include_tests": true,
  "include_docs":  false
},
"read_exclusions_inherit": ["conventions.secrets_locations", "conventions.vendored"]

Given the repo's topology: 

# target repo docs/REPO-TOPOLOGY.md fragment
modules:
  parser:
    code:  src/parser/
    tests: tests/parser/

commands:
  test: pnpm test

conventions:
  secrets_locations: [".env*", "secrets/"]
  vendored:          ["node_modules/", "vendor/"]

The launcher derives:

required_paths:
  rw:   [src/parser/, tests/parser/]
exit_criteria:
  - {type: command, value: "pnpm test"}
read_exclusions:
  - .env*
  - secrets/
  - node_modules/
  - vendor/

The operator reviews the derived mount plan in the AIDevOps UI before approval and can edit anything. The derivation is traceable: every mount line points back to the topology field that produced it.

Discovery from inside the sandbox

The agent does not grep around to find where things live. It asks Hermes: 

GET http://localhost:8765/v1/topology
→ {full resolved topology}

GET http://localhost:8765/v1/topology/module/parser
→ {code: "/workspace/src/parser/", tests: "/workspace/tests/parser/"}

GET http://localhost:8765/v1/topology/commands
→ {test: "pnpm test", lint: "pnpm lint"}

Flow 2 — Continuous status and events

sequenceDiagram
  participant AGENT as Agent
  participant HERMES as Hermes
  participant FS as ControlDir
  participant POLL as Poller
  participant UI as UI

  loop For each meaningful action
    AGENT->>HERMES: notify with type and payload
    HERMES->>HERMES: classify and audit
    HERMES->>FS: append event to events log
    HERMES->>FS: update status with phase step and heartbeat
  end

  loop Poll or MCP push
    POLL->>FS: read status and tail events
    POLL->>UI: stream updates
  end
Heartbeat: Hermes updates the status heartbeat at least every 10 seconds even if the agent is silent. If AIDevOps sees no heartbeat for 60 seconds it marks the run stalled.

Flow 4 — Completion and downstream workflow handoff

sequenceDiagram
  participant AGENT as Agent
  participant HERMES as Hermes
  participant FS as ControlDir
  participant POLL as Poller
  participant WE as WorkflowEngine
  participant TEST as TestRunner

  AGENT->>HERMES: notify tasks complete with manifest
  HERMES->>FS: append tasks_complete to events log

  POLL->>FS: read events
  POLL->>WE: emit RunCompleted

Separation of concerns: the agentic sandbox knows how to do one plan. Testing, review, and merging are workflow-engine concerns. The plan declares its exit_criteria; the workflow engine verifies them and drives the next step.

Both phases write the same schemas. Hermes writes status.json and events.jsonl in both modes; in Phase B, MCP carries the same JSON shapes live. Consumers that learn Phase A schemas do not break when Phase B lands. The schemas are the load-bearing contract.

Mount enforcement test plan

Seven tests to verify the enforcement stack actually holds on macOS plus Colima before Phase 5 is considered ready. Each test produces a pass or fail signal recorded against a known plan fixture.

IDTestExpectedLayer probed
T1Mount overlay: :ro base + :rw overlay; write to overlay subpathSucceedsLayer 2 mount mode
T2Mount overlay: write to base path outside overlayFails with EROFSLayer 2 mount mode
T3Sparse checkout: list /workspace/lib/ where lib/ was excludedNo such directoryLayer 1 visibility
T4Hermes refusal: agent invokes write tool against path inside worktree but NOT in required_paths.rwHermes refuses, logs to events, returns errorLayer 3 application refusal
T5Sudo and setuid: aiagent attempts sudoCommand not found or permission deniedLayer 4 container hardening
T6chmod attempt: aiagent attempts chmod 777 on a :ro mountEither chmod fails or write still fails because mount is read-onlyLayer 2 mount mode
T7Bind-remount escape: aiagent attempts mount -o remount,rw /workspacePermission denied (CAP_SYS_ADMIN dropped)Layer 4 capabilities
Run T1 and T2 first. They test the assumption that Docker bind-mount overlays correctly shadow on Colima. If they fail, the design falls back to Option B (enumerate every path individually as :ro or :rw) which is uglier but equivalent in effect.

Open questions

  • Multi-operator response handling: if two operators answer the same ask message, last-write-wins or reject? Defer.
  • Hermes failure semantics: if Hermes crashes mid-run, restart and resume or abort? Lean toward resume (status and events give recoverable state).
  • Outbox and inbox garbage collection: keep forever or archive? Defer.
  • Domain-specific OS users (LCM-Sandbox TODO 105): independent of this design but should align eventually.
  • Mount overlay vs explicit per-path mounts: verify overlay behavior on Colima before committing (test T1 and T2 above).
