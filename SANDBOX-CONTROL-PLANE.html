Agentic Sandbox Control Plane

How AIDevOps controls, observes, and reacts to work happening inside an agentic sandbox. Pairs with SANDBOX-ARCHITECTURE.md (what runs where) and SANDBOX-ORCHESTRATION.md (live transport).
Draft 2026-06-18 — design discussion converged; schemas not yet final.

Overview

The agentic sandbox accepts a plan from AIDevOps, executes it under bounded permissions, and reports status and events back to AIDevOps continuously. AIDevOps can intervene (pause, cancel, answer the agent's questions); the sandbox can request (credentials, decisions, operator input). When the agent finishes, the sandbox emits a completion event with a manifest; AIDevOps then drives the next workflow step (testing, review, merge).

Key design choice: AIDevOps never talks to the agent directly. It talks to Hermes, which runs inside the sandbox image alongside the agent. Hermes is the content layer (LLM routing, classification, audit) AND the control-plane endpoint (status, events, intervention). The agent talks to Hermes locally; Hermes is the only thing crossing the container boundary.

Mounts & files

The container has two bind mounts. Keeping them separate makes the security boundary explicit and lets the audit substrate outlive the worktree.

MountHost pathContainer pathModeOwnerPurpose
Workspace~/.sandbox-worktrees/sandbox-<id>-<ts>//workspacerw + ACLagentThe code the agent works on. Commits land here. ACL restricts writes to allowed_paths.
Control~/.lcm-sandbox/runs/<sandbox-id>//controlrwHermes onlyPlan delivery in, status/events out, inbox/outbox for bidirectional control.

Layout of /control

/control/
├── plan/
│   ├── plan.json         ← structured plan (delivered by AIDevOps before launch)
│   ├── plan.md           ← human-readable plan body (referenced by plan.json)
│   └── inputs/           ← any files the plan needs (test fixtures, configs, …)
├── status.json           ← Hermes writes continuously: current phase, step, heartbeat
├── events.jsonl          ← append-only audit log of every notable event
├── inbox/                ← AIDevOps drops messages here for Hermes to consume
│   └── <msg-id>.json
├── outbox/               ← Hermes drops messages here for AIDevOps to pick up
│   └── <msg-id>.json
└── hermes/
    ├── llm-calls.jsonl   ← per-call audit (model, classification, tokens)
    └── policy-log.jsonl  ← policy decisions (allowed, denied, transformed)
Why two mounts, not one? The worktree is the work surface — anything written there is reviewable and potentially commits to the repo. Control state (plan, status, events, operator messages) shouldn't pollute commits and shouldn't share the worktree's lifecycle. Keeping them separate also lets AIDevOps read live control state without walking the worktree.

Worktree lifecycle & mount selection

When the worktree is created, by whom, pinned to what

The worktree is created by lcm-sandbox Phase 1 (lcm_sandbox/core/worktree.py), after AIDevOps writes the plan to the host control directory and before docker launch. It is pinned to the source_revision the plan declares — typically the commit the plan was approved against, not HEAD of the parent repo. This means the agent works on a frozen snapshot, regardless of what the developer is currently editing in the parent repo.

What from the worktree is exposed in the container, and who decides

In v1, the whole worktree directory is bind-mounted at /workspace. Selectivity is enforced not by the mount but by an access-control list applied inside the container, declared by the plan: 

// plan.json (excerpt)
"context": {
  "repo":             "/workspace",
  "branch":           "feature/auto-todo-88",
  "source_revision":  "abc123def",
  "allowed_paths": {
    "write": ["src/", "tests/"],
    "read":  ["*"]
  }
}
DeclarationEffect in v1Where enforced
allowed_paths.writeAgent can only modify files under these paths in /workspace. Writes elsewhere fail with EACCES.ACL applied at container start by apply_agent_profile on the container's aiagent user; Hermes also enforces at the application layer (refuses write-tool calls outside this set).
allowed_paths.readInformational in v1 — agent can read anything under /workspace (whole worktree mounted).Not enforced as restriction; future Phase 2 hardening (TODO #107 in LCM-Sandbox) would apply git sparse-checkout to the worktree so only declared read paths are even visible.
Anything outside /workspaceInvisible to the agent (e.g., the parent repo, other worktrees, the user's home).Container isolation; the worktree is the only repo content mounted.
/control/plan/Plan files visible to Hermes only; the agent reaches them via Hermes' local HTTP endpoint.Hermes-only mount path; agent has no /control mount.

Why mount the whole worktree even with restrictive writes? Because the agent often needs to read code outside its write scope to understand context (e.g., touching src/ while reading lib/ for type signatures). Restricting reads via sparse checkout (TODO #107 in LCM-Sandbox) is a hardening pass for cases where threat model requires it — e.g., the plan covers third-party code the agent should not be able to exfiltrate. Not v1.

Future option: bind individual subdirectories instead of the whole worktree

Some scenarios may want to mount only specific worktree subdirectories at named container paths — e.g., mount worktree/services/api/ at /workspace and worktree/libs/shared/ at /aux. This is not v1; the simpler "whole worktree at /workspace + ACL" model handles the current use cases. If this becomes necessary, the plan would declare named mount-mappings instead of (or alongside) allowed_paths, and docker_launcher.py would assemble -v flags accordingly. Track as a deferred enhancement.

What the agent receives

The agent does not read plan.json directly. It calls a local Hermes endpoint: 

GET http://localhost:8765/v1/plan
→ {
  "title": "Implement TODO #88",
  "instructions": "<rendered markdown>",
  "exit_criteria": [...],
  "permitted_actions": ["read", "write within allowed_paths", "commit", "run tests"],
  "ask_operator_endpoint": "POST http://localhost:8765/v1/ask"
}

This uniform interface means the agent behaves identically whether the plan came from AIDevOps, a CLI test run, or a future trigger source. Hermes also redacts anything the agent shouldn't see (operator credentials, internal IDs).

Flow 2 — Continuous status & events

sequenceDiagram
  participant AGENT as Agent
  participant HERMES as Hermes (in container)
  participant FS as /control on host
  participant POLL as AIDevOps poller
  participant UI as Operator UI

  loop For each meaningful action
    AGENT->>HERMES: notify({type: file_edited, path: src/foo.ts})
    HERMES->>HERMES: classify, audit
    HERMES->>FS: events.jsonl += {ts, type, payload, seq}
    HERMES->>FS: status.json = {phase, step, last_action, heartbeat}
  end

  loop Polling (every N seconds, or MCP push when live)
    POLL->>FS: read status.json + tail events.jsonl
    POLL->>UI: stream updates
  end
Heartbeat: Hermes updates status.json.heartbeat at least every 10s even if the agent is silent. If AIDevOps sees no heartbeat for 60s it marks the run stalled and surfaces it. This catches hung agents and dead containers.

Flow 3 — Agent blocked, needs operator input

sequenceDiagram
  participant AGENT as Agent
  participant HERMES as Hermes
  participant FS as /control on host
  participant POLL as AIDevOps poller
  participant OP as Operator (UI)

  AGENT->>HERMES: POST /v1/ask {question, options, context}
  HERMES->>FS: outbox/<msg-id>.json = {type: ask, question, options, context}
  HERMES->>FS: status.json.blocked_on = <msg-id>
  HERMES->>FS: events.jsonl += {type: blocked, msg_id}
  POLL->>FS: read outbox
  POLL->>OP: surface question in UI
  OP->>POLL: select answer
  POLL->>FS: inbox/<msg-id>.json = {type: answer, value}
  HERMES->>FS: read inbox/<msg-id>.json
  HERMES->>FS: status.json.blocked_on = null
  HERMES->>FS: events.jsonl += {type: unblocked, msg_id, answer}
  HERMES->>AGENT: 200 {answer}
  AGENT->>AGENT: continue execution

The agent's POST /v1/ask call blocks until Hermes resolves it. Timeout is configurable per plan. If the operator doesn't answer within the timeout, Hermes returns an error to the agent which can then emit a failed: timed_out_on_input event and exit cleanly.

Flow 4 — Completion → downstream workflow trigger

sequenceDiagram
  participant AGENT as Agent
  participant HERMES as Hermes
  participant FS as ControlDir
  participant POLL as Poller
  participant WE as WorkflowEngine
  participant TEST as TestRunner

  AGENT->>HERMES: notify tasks complete with manifest
  HERMES->>FS: append tasks_complete to events log
  HERMES->>FS: update status to complete
  POLL->>FS: read events
  POLL->>WE: emit RunCompleted
  WE->>WE: consult workflow for next step
  WE->>FS: write next plan into runs dir
  WE->>TEST: launch test runner
  Note over WE,TEST: Test runner can be another sandbox reading the worktree
  TEST->>WE: report test result
  alt tests pass
    WE->>WE: advance to review or merge step
  else tests fail
    WE->>WE: respawn agentic sandbox with follow up plan
    Note over WE: follow up plan carries the failing test output
  end
Separation of concerns: the agentic sandbox knows how to do one plan. It does not know about testing, review, or merge. Those are workflow-engine concerns. The plan declares its exit_criteria; the workflow engine verifies them and drives the next step.

Flow 5 — Failure

sequenceDiagram
  participant AGENT as Agent
  participant HERMES as Hermes
  participant FS as /control on host
  participant POLL as AIDevOps poller
  participant WE as AIDevOps Workflow Engine

  alt Agent reports failure
    AGENT->>HERMES: notify({type: failed, reason, error, partial_progress})
    HERMES->>FS: events.jsonl += {type: failed, reason, error}
    HERMES->>FS: status.json = {phase: failed, reason}
  else Container dies unexpectedly
    Note over HERMES,FS: status.json heartbeat stops updating
    POLL->>POLL: detect heartbeat > 60s old
    POLL->>WE: emit RunStalled event
  else Hermes itself crashes
    Note over HERMES,FS: no new writes to status.json or events.jsonl
    POLL->>POLL: detect heartbeat > 60s old
    POLL->>WE: emit RunStalled event
  end

  WE->>WE: workflow definition specifies failure action
  Note over WE: e.g. notify operator, archive artifacts,
spawn diagnosis sandbox, retry with adjusted plan

The audit substrate (worktree + /control on host) survives container death, so an operator can always go inspect what the agent did and what state it reached before failing.

Both phases write the same schemas. status.json and events.jsonl are written by Hermes in both modes; in Phase B, MCP carries the same JSON shapes live. Consumers that learn Phase A schemas do not break when Phase B lands. The schemas are the load-bearing contract.

Locked design decisions

#DecisionRationale
1Hermes runs inside lcm-hermes-agent:latestAlready designed as content/policy/audit layer; natural place to host the control endpoint; agent stays credential-free.
2AIDevOps talks to Hermes, not the agentClean boundary; agent shouldn't run HTTP servers; Hermes already does provider-side comms.
3Two container mounts: /workspace + /controlWork surface and audit substrate have different lifecycles, ownership, and review semantics.
4Plan delivered as files in /control/plan/ before launchSame medium as status; survives restart; AIDevOps writes once and forgets.
5Status + events as files in /control/ alwaysDurable audit substrate independent of transport. MCP later carries the same shapes live.
6Phase A transport: filesystem pollingShips now, no upstream blockers; correct as a fallback even after Phase B lands.
7Phase B transport: MCP Streamable HTTP + OAuth 2.1 per SANDBOX-ORCHESTRATION.mdAlready designed; blocked on upstream claude-code issues #28293 + #36665, not on architecture.
8SSH / Ghostty rejectedFights hardened container posture; live-debug use case is better served by mounting the same worktree into an interactive sandbox.
9Downstream workflow triggering is an AIDevOps Workflow-Engine concern, not a sandbox concernSandbox reports events faithfully; workflow engine decides what to do based on plan's exit_criteria.
10Schemas (plan.json, status.json, events.jsonl, inbox/outbox) are the load-bearing artifactBoth transports carry the same shapes; consumers don't break across the Phase A → B transition.

Open questions / deferred

  • Multi-operator response handling — what happens if two operators answer the same ask message? Last-write-wins, first-write-wins, or reject? Defer to v2.
  • Sparse checkout for read isolation — LCM-Sandbox TODO #107. Defer; v1 mounts whole worktree, relies on ACL for write isolation.
  • Hermes failure semantics — if Hermes crashes mid-run, can it restart and resume? Or does the run abort? Lean toward restart-and-resume (status.json + events.jsonl give recoverable state), but needs careful design of the agent's local connection logic.
  • Domain-specific OS users in sandbox — LCM-Sandbox TODO #105. Independent of this design but should align: per-domain user adds OS-layer audit beyond Hermes' application-layer audit.
  • Outbox/inbox garbage collection — resolved messages stay forever or get archived? Defer; trivial.
