In-Sandbox Agent Configuration

Visual explainer of how agent CLIs inside the sandbox container are prevented from hanging on permission prompts. Companion to SANDBOX-AGENT-CONFIG.md.

The problem in one picture

Inside the container there is no human to answer "are you sure?". Any prompt = wasted timeout. The job of the in-container config is to make sure the agent never sees a prompt.

Four layered defenses

The in-sandbox agent config is the fourth layer of defense. The other three give us hard guarantees, which is what makes it safe to let the agent config default to fully permissive.

Every prompt path the agent could hit

This is the decision flow inside Claude Code when the agent attempts an action. Each diamond is a place a prompt could fire — and we silence each one with a specific config key.

Two prompt paths bypass this entire flow — the rm -rf / and rm -rf ~ circuit breakers, and the "Overwrite existing file" dialog. These cannot be silenced from config alone. See Defensive shims.

Defensive shims (the belt for the suspenders)

A. rm-shim

Mitigates rows 8 and 9 (circuit-breaker prompts).

Install /usr/local/bin/rm as a wrapper that refuses rm -rf /, rm -rf ~, rm -rf $HOME directly. Claude Code never sees those commands → its circuit breaker never fires.

#!/bin/bash
# /usr/local/bin/rm (precedes /bin/rm in PATH)
case " $* " in
  *" -rf /"*|*" -rf /*"*|*" -rf ~"*|*" -rf $HOME"*|*" -rf $HOME/"*)
    echo "rm-shim: refused dangerous pattern: $*" >&2
    exit 1 ;;
esac
exec /bin/rm "$@"

B. stdin-EOF shim

Catches row 13 (unknown future prompts).

Wrap the agent invocation so stdin is closed (any read returns EOF) and there's no TTY. If anything slips past config, the agent fails in seconds instead of hanging for the full timeout.

exec setsid </dev/null \
  timeout --signal=TERM --kill-after=10s \
  ${TIMEOUT_MINUTES}m \
  agent-cli "$@"

C. Pre-flight smoke test

Catches config drift before the real run wastes the timeout budget.

Before handing off to the agent, the entrypoint runs a 30-second probe of each silenced path. If any probe hangs, abort with a clear error captured by Phase 6.

# /opt/lcm-sandbox/smoke-test.sh
set -e
timeout 30s claude --print "say ok" </dev/null     # Bash + WebFetch probe
echo "smoke test passed"

Open audit items

  1. Codex audit. Non-interactive mode? Auth via env var? Hooks/approval surfaces? Documented answer required.
  2. Gemini audit. Same set of questions.
  3. Confirm rm-shim doesn't break pip/npm cache cleanup. Probe via a real pip install + npm install inside the image.
  4. Reconcile disableAllHooks: true with desired in-container audit hooks. Phase 2 may want a single non-prompting hook for audit logging.
  5. Smoke test scope. Should the smoke test probe a Bash command, an Edit, and a WebFetch — or is the single --print probe sufficient? Recommend probing all four.
